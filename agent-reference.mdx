## Authentication

All requests require the `X-Sim-Api-Key` header. Get your key from the [Sim Dashboard](https://sim.dune.com).

```bash
curl -X GET "https://api.sim.dune.com/v1/evm/balances/{address}" \
     -H "X-Sim-Api-Key: YOUR_API_KEY"
```

- All requests must use **HTTPS**. Plain HTTP will fail.
- Never expose API keys in client-side code or public repositories.

## Base URL

```
https://api.sim.dune.com
```

Stable endpoints use `/v1/`. Beta endpoints use `/beta/`.

|---|---|---|---|

## Pagination

All list endpoints use **cursor-based pagination** with the same pattern:

1. Set `limit` to your desired page size.
2. The response includes `next_offset` when more results exist.
3. Pass `next_offset` as the `offset` query parameter in your next request.
4. When `next_offset` is absent or `null`, you have reached the last page.

<Warning>
**Never construct your own `offset` value.** The offset is an opaque cursor — only use the value returned by the API in `next_offset`.
</Warning>

```bash
# First page
curl "https://api.sim.dune.com/v1/evm/balances/{address}?chain_ids=1,8453&limit=100" \
     -H "X-Sim-Api-Key: YOUR_API_KEY"

# Next page (use next_offset from previous response)
curl "https://api.sim.dune.com/v1/evm/balances/{address}?chain_ids=1,8453&limit=100&offset=NEXT_OFFSET_VALUE" \
     -H "X-Sim-Api-Key: YOUR_API_KEY"
```

## Error Handling

### HTTP Error Codes

| Status | Description | Action |
|---|---|---|
| 400 | Malformed request | Check address format, parameter names, and values |
| 401 | Invalid or missing API key | Verify `X-Sim-Api-Key` header |
| 402 | Compute unit quota exceeded | Upgrade plan. Do not retry |
| 404 | Resource not found | Check endpoint URL and path parameters |
| 429 | Rate limited | Implement exponential backoff, then retry |
| 500 | Server error | Retry after a short delay |

### Error Response Format

```json
{ "error": "Description of what went wrong" }
```

<Note>
The error property can be either `"error"` or `"message"` depending on the error type.
</Note>

### Warnings (Non-Fatal)

A `200` response can include a `warnings[]` array alongside valid data. This occurs when some requested `chain_ids` are unsupported — the API returns data for supported chains and warns about the rest.

```json
{
  "balances": [...],
  "warnings": [
    {
      "code": "UNSUPPORTED_CHAIN_IDS",
      "message": "Some requested chain_ids are not supported.",
      "chain_ids": [9999],
      "docs_url": "https://docs.sim.dune.com/evm/supported-chains"
    }
  ]
}
```

Always check for and surface warnings to the user.

## Common Workflows

### 1. Portfolio Balance Lookup

Fetch all token balances for a wallet across specific chains, paginating to completion.

```bash
# Page 1
curl "https://api.sim.dune.com/v1/evm/balances/0xd8da6bf26964af9d7eed9e03e53415d37aa96045?chain_ids=1,8453,42161&limit=500&exclude_spam_tokens=true" \
     -H "X-Sim-Api-Key: YOUR_API_KEY"

# If response contains next_offset, fetch page 2:
curl "https://api.sim.dune.com/v1/evm/balances/0xd8da6bf26964af9d7eed9e03e53415d37aa96045?chain_ids=1,8453,42161&limit=500&exclude_spam_tokens=true&offset=NEXT_OFFSET_VALUE" \
     -H "X-Sim-Api-Key: YOUR_API_KEY"
```

Sum `value_usd` across all pages for total portfolio value.

### 2. Activity Feed with Filters

Get recent ERC-20 receives for a wallet.

```bash
curl "https://api.sim.dune.com/v1/evm/activity/0xd8da6bf26964af9d7eed9e03e53415d37aa96045?chain_ids=1,8453&activity_type=receive&asset_type=erc20&limit=50" \
     -H "X-Sim-Api-Key: YOUR_API_KEY"
```

Filter combos: `activity_type` accepts comma-separated values (`send,receive`). Same for `asset_type` and `token_address`.

### 3. Token Metadata & Price Lookup

Get current and historical price for a token.

```bash
curl "https://api.sim.dune.com/v1/evm/token-info/0xa0b86991c6218b36c1d19d4a2e9eb0ce3606eb48?chain_ids=1&historical_prices=720,168,24" \
     -H "X-Sim-Api-Key: YOUR_API_KEY"
```

Returns current `price_usd` plus `historical_prices[]` array with prices at 720h, 168h, and 24h ago. Compute percent changes yourself.

### 4. Webhook Subscription Setup

Create a webhook to receive real-time balance changes.

```bash
curl -X POST "https://api.sim.dune.com/v1/evm/subscriptions" \
     -H "X-Sim-Api-Key: YOUR_API_KEY" \
     -H "Content-Type: application/json" \
     -d '{
       "webhook_url": "https://your-app.com/webhook",
       "type": "balances",
       "chain_ids": [1, 8453],
       "addresses": ["0xd8da6bf26964af9d7eed9e03e53415d37aa96045"]
     }'
```

Validate incoming webhooks using the `dune-webhook-signature` header. Handle retries via `dune-webhook-retry-index` (0 = first attempt, up to 5 retries over 24h).

## Rules for Agents

### Always

- Include the `X-Sim-Api-Key` header in every request
- Use HTTPS
- Pass explicit `chain_ids` for predictable CU costs
- Handle `next_offset` for pagination — loop until absent/null
- Check for `warnings[]` in 200 responses
- Implement exponential backoff for 429 and 500 errors

### Ask First

- Before omitting `chain_ids` (uses the `default` tag, queries 20+ chains)
- Before using beta endpoints (`/beta/`) in production code
- Before creating webhook subscriptions (ongoing CU cost per event)

### Never

- Construct your own `offset` values — only use `next_offset` from the API
- Use chain names (e.g., "ethereum") in the `chain_ids` parameter — use numeric IDs or tags only
- Assume all endpoints share the same parameters or response shapes
- Hardcode API keys in client-side code
- Expect write/transaction-submission capabilities — Sim APIs are read-only (except subscription management)

## What This API Does NOT Support

- **No write/POST endpoints** — Read-only API (except Subscriptions management). No transaction submission, token transfers, or contract interactions.
- **No historical balance snapshots** — `historical_prices` returns past token prices, not past balance amounts.
- **No real-time WebSocket streaming** — Use REST polling or webhook Subscriptions for real-time data.
- **No ENS / domain resolution** — Addresses must be raw hex (EVM) or base58 (SVM). Resolve ENS names before calling the API.
- **No aggregated EVM + SVM in one call** — EVM (`/v1/evm/`, `/beta/evm/`) and SVM (`/beta/svm/`) are separate namespaces.
- **No custom SQL / query execution** — Sim APIs are not Dune's Query API. They do not execute arbitrary SQL.
- **No token swap / DEX routing** — Provides balances and positions, not trading functionality.
- **No chain names in `chain_ids`** — Only numeric chain IDs or tags (`default`, `mainnet`, `testnet`).

- Include X-Sim-Api-Key header
- Use HTTPS
- Pass explicit chain_ids
- Handle next_offset pagination
- Check warnings[] in 200 responses
- Exponential backoff on 429/500

ASK FIRST:
- Omitting chain_ids (default tag = 20+ chains)
- Using /beta/ endpoints in production
- Creating webhooks (ongoing CU cost)

NEVER:
- Construct your own offset values
- Use chain names in chain_ids (numeric IDs or tags only)
- Assume endpoints share parameters/response shapes
- Hardcode API keys in client-side code
- Expect write/transaction capabilities (read-only except subscriptions)

## What This API Does NOT Support

- No write/POST endpoints (read-only, except subscription management)
- No historical balance snapshots (historical_prices = past prices, not past balances)
- No WebSocket streaming (use REST polling or webhooks)
- No ENS/domain resolution (resolve names before calling)
- No combined EVM + SVM in one call (separate namespaces)
- No custom SQL/query execution (not Dune Query API)
- No token swap/DEX routing
- No chain names in chain_ids parameter
